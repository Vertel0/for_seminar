# solution — эталон семинара Л4 (для семинариста)

Три версии клиента по слотам + self-contained демо + тесты. Запускается без поднятия
сервера руками (intake гоняется в процессе через ASGI-транспорт).

## Запуск

```bash
python3 run_demo.py      # наглядный контраст: наивный теряет данные, надёжный — нет
python3 -m pytest        # 4 теста (см. ниже)
```

## Файлы

- `client_naive.py` — слот 1: последовательный, без ретраев. Под вредностью **теряет** данные.
- `client_parallel.py` — слот 2: семафор на K + батчинг (`/telemetry/batch`).
- `client_reliable.py` — слот 3: таймаут + ретрай (exp backoff+jitter) + идемпотентность
  (стабильный ключ) + локальная очередь (раунды до-отправки). **0 потерь, 0 задвоений.**
- `run_demo.py` — прогон контраста.
- `tests/test_clients.py` — проверки.

## Что показывает демо

```
НАИВНЫЙ под вредностью (fail_rate=0.2):     потеряно ~90 из 500
НАДЁЖНЫЙ под вредностью + ack-loss + падение intake на 1с:
        принято уникальных: 500 / 500
        потерь = 0, задвоений = 0  (повторов задедуплено: ~90)
```

То есть надёжный клиент не только ничего не теряет, но и не задваивает: повторы случались
(ack-loss), и их **спасла идемпотентность** — сервер дедупнул по ключу.

## Тесты

- `test_naive_delivers_all_when_healthy` — на «здоровом» intake наивный доносит всё.
- `test_parallel_batch_delivers_all_when_healthy` — батчинг доносит всё и **сокращает число запросов**.
- `test_reliable_zero_loss_under_failure_and_outage` — под `fail_rate=0.3` + падением на 0.7с: **0 потерь**.
- `test_idempotency_prevents_double_write_under_ack_loss` — под `ack_loss_rate=0.4`: не задвоилось,
  но дедуп **сработал** (`duplicates>0`).

## Мораль для зала

Надёжность — это поведение **под отказом**. Таймаут + умный ретрай + идемпотентность +
локальная очередь = клиент переживает временную смерть зависимости. Это прямо модуль их
итогового проекта (приём телеметрии).
